# A1 — Capture fallback tool output bytes

**Priority:** P0 for v3.4 Block A — foundational for all token-economy work
**Plan reference:** v3.4 Block A (roadmap revision after the 2026-05-12 field-test re-run)

## Goal

Close the telemetry gap surfaced by the v3.4 token-economy analysis: the
session-tracking hook records `input_size` for every fallback tool call
(Read, Bash, Edit, …) but **not** `output_size`. Without the output
half, we can compute exactly half of a session's byte budget — myco
output is in `telemetry.jsonl`, fallback output is invisible.

After this ticket: every `*_external.jsonl` row carries both
`input_size` and `output_size`, and `myco session export` totals them
per tool.

## What Claude Code already gives us

The Claude Code PostToolUse hook spec already pipes a `tool_response`
field in the JSON payload alongside `tool_name` and `tool_input`. Our
parser at `internal/telemetry/external.go:37-42` explicitly ignores it
("We don't use tool_response"). Removing that explicit ignore is the
core change.

The exact shape of `tool_response` varies by tool:
- Read: `{"file": {"contents": "..."}}` (the file bytes the agent saw)
- Bash: `{"stdout": "...", "stderr": "...", "interrupted": false}`
- Edit / Write: a small confirmation object (~few hundred bytes)
- WebSearch / WebFetch: result list

We don't need to parse the shape — `output_size = len(tool_response)`
is the byte count we care about. The raw JSON bytes the model has to
process equate one-to-one with token cost.

## Critical files

- `internal/telemetry/external.go`:
  - Extend `postToolUsePayload`: add `ToolResponse json.RawMessage`.
  - Extend `ExternalRecord`: add `OutputSize int \`json:"output_size"\``.
  - `ParsePostToolUse` populates `OutputSize = len(p.ToolResponse)`.
- `internal/telemetry/external.go` `SummarizeExternal` — extend
  `ExternalSummary` with `InputBytes` + `OutputBytes` totals, surfaced
  per tool/category.
- `cmd/myco/main.go` (`session export`) — display the new per-tool
  byte columns alongside `calls`.
- New test cases in `internal/telemetry/telemetry_test.go` covering:
  - PostToolUse payload with tool_response → OutputSize populated
  - PostToolUse payload without tool_response (legacy / unknown
    tools) → OutputSize = 0, no error
  - Aggregator sums OutputSize per tool

## Acceptance criteria

- `task check` passes.
- A live PostToolUse hook fired by Claude Code populates `output_size`
  in the session JSONL (verifiable by `cat .mycelium/session_*_external.jsonl | head | jq .output_size`).
- `myco session export <id>` shows a new `out_bytes` column for each
  fallback tool, plus a session-total row.
- Legacy session files (without `output_size`) still parse — fall
  back to 0 silently.

## Why this ships first

A2 (session cost) and A3 (counterfactual savings) both depend on
having the fallback output bytes. This is the 1-day foundational
piece.

## Notes

- Don't try to parse Read's `tool_response` to extract just the file
  content's bytes. The raw JSON is what flows back to the model as
  context, so its full length is the token cost.
- Privacy: the JSON bytes are already on disk inside the agent's
  process; this ticket only counts their length. No new content
  retention.
