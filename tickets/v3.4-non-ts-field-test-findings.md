# v3.4 prerequisite — non-TS field test: Go dogfooding findings

**Status:** preliminary signal, telemetry-incomplete.
**Plan reference:** `~/.claude/plans/7-v3.1-broader-hyphae.md` (v3.4 section).
**Goal:** unlock v3.4 (route literals + new languages) by validating that
mycelium works on non-TS codebases and surfacing the gaps worth filling.

## What this is

The v3.1 plan gates v3.4 ("more languages + route literals") on a
non-TS field test, because every prior field datum was a TS monorepo
(`monorepo-4`) and that's not enough to drive language prioritisation
or confirm the route-literal pattern generalises.

The natural first non-TS field test target was **mycelium itself** — a
~25k-LOC Go codebase, fully indexed, and the agent (me, Claude Code
with mycelium MCP wired) had just spent 11+ hours implementing
v3.1.2 + v3.3 on it. That's the *dogfooding* case the CLAUDE.md
already documents.

## Data captured

Telemetry was **off** in `.mycelium.yml` for the duration of the test,
so per-session MCP-tool counts are missing. Hook-recorded
fallback-tool counts are intact (the session-tracking hook in
`.claude/settings.json` writes them to
`.mycelium/session_*_external.jsonl` regardless of telemetry).

`.mycelium.yml` now sets `telemetry.enabled: true` — future

## Qualitative findings

Tagged like the v3.1 plan: **[strong]** = directly seen in
transcripts. **[medium]** = inferred from per-session shape.
**[speculative]** = generalising beyond this data.

### G1. `Bash/grep` is still the agent's default for literal hunts [strong]

54 grep calls today, second after `Edit`. Every one of those was a
case where `find_symbol` (for a name) or `search_lexical` (for a
literal) would have done the same job. The agent (me) reached for
grep when:
- counting things ("how many tests pass?", "how many sessions today?")
- searching JSON / config / SQL output that's not part of the code
  graph (e.g. recent commits, go.mod entries before B3 indexed them)
- one-shot greps where typing `bash grep` was faster than composing a
  myco call with the right project filter

Implication: the v3.1.2 priming work cut Read-driven exploration but
didn't dent the "grep for one-shot hits" reflex. Either (a) myco
needs a literally-faster surface for one-shot hits (read-only,
no daemon round-trip), or (b) some grep usage is genuinely fine and
the metric was wrong to count it.

### G2. `read_focused` still under-used [strong]

83 `Read` calls, **0** observed `read_focused` calls. Every file read
in this session that exceeded ~50 lines was a candidate for
`read_focused` — opening a parser file, inspecting a doctor check,
re-reading a migration. None used it.

Same shape as monorepo-4's F5. The v3.1.2 tool-description rewrite
helped agents pick the *right path-relative-form*; it didn't yet
push them to prefer `read_focused` over `Read`. v3.2's CLAUDE.md
priming explicitly mentions `read_focused` but the agent's default
when given a known path is still `Read`.

### G3. Routes / config keys don't exist in this codebase [medium]

mycelium has **no equivalents** of monorepo-4's F3 (i18n JSON) and F4
(route literals): no externalised strings, no
`createFileRoute(...)`-style framework registration. Cobra
subcommands (`cmd.AddCommand(...)`) are Go function calls already
findable via `find_symbol`. So v3.3's documents surface and v3.4's
route-literal idea don't validate on this codebase. They need a
different field test (Django, Flask, Spring, Axum, …) to confirm
the patterns generalise.

### G4. Go-specific friction is small [medium]

`Bash/go` (46 calls) is all test/build/vet — unavoidable, not a
mycelium gap. `find_symbol` worked first-try on every Go symbol I
threw at it (`UpsertFile`, `runDocuments`, `loadResolvers`, etc).
No `null`-result failures of the kind that drove the v3.1 A1 fix.

Implication: Go support is *production-quality*. Whatever
language gaps motivate v3.4 lie elsewhere.

### G5. Telemetry-off was a real adoption defect [strong]

I was about to write up "the agent made X myco calls vs Y
fallbacks" — but `myco session list` shows `calls: 0` for every
session because `telemetry.enabled` defaulted to false. The
session hook recorded only the fallback side. For dogfooding to
produce useful field-test data, telemetry needs to be on by
default in dev/dogfood repos — or `myco init` needs to nudge
harder. The v3.1 setup wizard does ask, but I'd skipped wizard
setup.

Concrete v3.4 implication: add a doctor warning *"telemetry is
off; you're flying blind on adoption metrics"* (gated to dev
repos / when `.mycelium/sessions/` shows traffic).

## What this means for v3.4 prioritisation

The v3.1 plan listed two v3.4 themes:

1. **A2 — route literals as a symbol kind.** No data here either
   way. mycelium's Go codebase has no framework-route pattern.
   Need a Python/Django or Rust/Axum field test.

2. **Languages priority (Rust / Java-Kotlin / C-C++ / Ruby / PHP /
   C# / Swift).** Today's data doesn't move the needle. Go is
   solid; the next language depends on the next field-test repo.

The strongest signal today isn't actually v3.4 — it's that v3.1.2
priming and v3.3 documents both work but the agent's reflexes
(grep, Read-not-read_focused) haven't shifted as much as the
tool-description rewrite hoped. That's a v3.5 / v4 candidate:
**adoption-fixed-point** work that targets the residual fallbacks.

## Recommended next step

Pick a real non-TS, non-Go codebase for a proper v3.4 prereq field
test. Three candidates, ranked:

1. **A Python/Django repo (mid-size, ~5-15k LOC).** Exercises
   route literals (`path("/foo", view)` in `urls.py`), template
   indexing (analogous to i18n JSON), and validates the Python
   parser/resolver under real-world load.
2. **A Rust/Axum or Actix repo.** Exercises route literals
   (`Router::new().route("/foo", ...)`) and is the first non-Go
   *systems* language. Highest value for language prioritisation.
3. **A Java/Spring repo.** Spring's `@RequestMapping("/foo")`
   annotation hits the route-literal pattern from a different
   angle.

Each test follows the v3.1.0 template: open the repo with myco
wired in, attempt a real coding task (~30 min), export the
session, compute the same metrics, file findings as a `Fn` block
in this ticket's successor.
