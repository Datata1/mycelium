# A2 — Session cost aggregator (with-myco baseline)

**Priority:** P1 for v3.4 Block A — turns A1's bytes into a model-token estimate
**Depends on:** [v3.4-a1-fallback-output-telemetry.md](v3.4-a1-fallback-output-telemetry.md)

## Goal

Answer the user's first long-term question: **how expensive was a
session with myco?**

After this ticket: `myco session export <id>` (and the markdown form)
includes a `cost estimate` block — total bytes, split into myco vs.
fallback, converted to an estimated token count using a configurable
chars-per-token ratio. The number is approximate but **directional**:
across many sessions it tells the user whether myco usage is
trending heavier or lighter.

## What it computes

For each session:

```
cost estimate
  myco_output_bytes:       138.3 KiB    (tool→agent)
  myco_input_bytes:          3.5 KiB    (agent→tool)
  fallback_output_bytes:    ~160 KiB    (from A1)
  fallback_input_bytes:      ~5 KiB
  total_bytes:              ~307 KiB
  estimated_tokens:        ~78,500      (at 4 chars/token)
  per_tool:
    find_symbol             38.5 KiB / 9,800 tok
    read_focused            84.9 KiB / 21,700 tok
    Read (fallback)        ~150 KiB / ~38,400 tok
    Bash/grep (fallback)    ~10 KiB / ~2,600 tok
    ...
```

The chars-per-token ratio is the only knob. Default 4 (typical for
English-heavy code+JSON output with Claude's tokenizer). Configurable
in `.mycelium.yml` under `telemetry.chars_per_token` for users who
benchmarked against their own tokenizer.

## Critical files

- `internal/telemetry/aggregate.go` — new `SessionCost` type and
  `ComputeSessionCost(myco []ToolSummary, fallback []ExternalSummary, charsPerToken float64) SessionCost`.
- `internal/config/config.go` — `TelemetryConfig.CharsPerToken float64`
  (default 4.0 via `Default()`).
- `cmd/myco/main.go` — `session export` reads the config, calls
  `ComputeSessionCost`, prints the cost block (markdown + table forms).
- New tests in `internal/telemetry/telemetry_test.go`:
  - Cost computed from synthetic summaries
  - Default ratio applied when config missing
  - Per-tool breakdown sums to the session total

## Acceptance criteria

- `task check` passes.
- `myco session export <id> --format markdown` shows a `cost
  estimate` section with bytes + estimated tokens.
- Across multiple sessions, `myco session list --cost` (new flag)
  shows the per-session estimated token spend so trends are
  visible without exporting each one.
- The chars-per-token ratio is overridable via config and CLI flag.

## What this enables

Once shipped, a user can ask: *"is my agent's myco usage trending
toward 100k or 10k tokens per session?"* and get a directional
answer. Combined with v3.1.2's adoption telemetry, this is the first
honest "is myco worth its keep" measurement.

The number is **NOT** a billing-accurate token count — it's a byte
proxy. For billing-accurate counts a user would have to instrument
the agent's actual LLM-side tokenizer. We're explicitly trading
accuracy for visibility.

## Out of scope (defer to A3)

The *counterfactual* — what would the same session have cost
**without** myco — is A3's job. This ticket only reports the
actually-incurred cost.
