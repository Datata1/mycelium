# A3 — Counterfactual cost model (without-myco estimation)

**Priority:** P1 for v3.4 Block A — answers the second long-term question
**Depends on:** [v3.4-a1-fallback-output-telemetry.md](v3.4-a1-fallback-output-telemetry.md), [v3.4-a2-session-cost-aggregator.md](v3.4-a2-session-cost-aggregator.md)

## Goal

Answer the user's second long-term question: **how expensive would the
same session have been *without* myco?**

After this ticket: every myco call in `telemetry.jsonl` gets paired
with a `counterfactual_bytes` estimate — the byte size of the
equivalent fallback operation (`grep`, `Read`, …) that the agent
would have run if myco weren't available. The aggregator sums them,
A2's cost block grows a `without_myco_estimate` row, and the
difference is the headline "myco saved/cost N tokens" number.

This is the **A/B test** the user asked for, modelled rather than
live (running real `grep` calls in parallel during the live session
would 10× the latency and is the v4 move once the data justifies it).

## The model

Per myco-tool, the counterfactual is:

| myco tool | counterfactual operation | byte estimate |
| --- | --- | --- |
| `find_symbol{name}` | `grep -rn 'name' src/` | hits × ~120 B/hit (line + path + context) |
| `read_focused{path,focus}` | `Read{path}` | `files.size_bytes` for that path |
| `get_references{target}` | `grep -rn 'target' src/` (textual fallback) | (resolved + textual) × ~120 B |
| `search_lexical{pattern}` | `grep -rn 'pattern' src/` | ~ same as actual output (myco ≈ grep here) |
| `get_file_outline{path}` | `Read{path}` then mental parse | full file size |
| `get_file_summary{path}` | `Read{path}` | full file size |
| `get_neighborhood{target,depth}` | iterated `grep` + `Read` (model as `find_symbol × 3 hops` worth of bytes per result node) | nodes × ~600 B |
| `impact_analysis{target}` | iterated `grep -rn` for ALL callers | hits × ~120 B |
| `critical_path{from,to}` | not approximated — log as `nil` and exclude | — |
| `find_document_key{key}` | `grep -rn 'key' i18n/ package.json go.mod` | hits × ~120 B |

The constants are **calibrated**, not invented: a small bench during
implementation runs each grep variant against the mycelium self-index
and records the actual output size. The 120-B and 600-B figures above
are rough placeholders to replace with measured medians.

### Why modelled, not live

Live counterfactual (actually running `grep` for every `find_symbol`
call) would:
- Add 1-5 seconds latency per myco call → kills the v2.4 speed promise.
- Re-traverse every file on every search → no caching gain from the index.
- Hit `find`/`grep` on huge monorepos uncached → contend with the
  user's editor / build.

Modelled is good enough at v3.4 scale. A v4 "live A/B mode" flag
(`--bench`) could opt-in to the slow path for one-off validation runs.

## Critical files

- `internal/telemetry/counterfactual.go` (new): per-tool counterfactual
  estimator. Takes a `ToolCall` (tool name, args, actual output size)
  and returns `(estimated_fallback_bytes int, ok bool)`.
- `internal/telemetry/telemetry.go` — `Recorder` writes the
  counterfactual estimate alongside each row. New field
  `counterfactual_bytes` in the JSONL.
- `internal/telemetry/aggregate.go` — `SessionCost` grows
  `WithoutMycoEstimate int` and `NetSavings int` (= cost - counterfactual).
- `cmd/myco/main.go` `session export` — extend the cost block:

  ```
  cost estimate
    with_myco_actual:        ~78,500 tokens
    without_myco_modeled:   ~155,000 tokens
    estimated_savings:       ~76,500 tokens   (~49%)
  ```

- `internal/telemetry/calibration_test.go` (new): runs the
  per-tool bench against the self-index to validate the constants
  haven't drifted (xfail if ratios change > 50%).

## Acceptance criteria

- `task check` passes.
- Every myco call in `telemetry.jsonl` written after this lands
  carries a `counterfactual_bytes` field.
- `myco session export <id>` shows `with_myco_actual`,
  `without_myco_modeled`, `estimated_savings` as a trio.
- The calibration bench prints the measured constants; the test
  fails loudly when they drift so the model stays honest.
- The savings number is presented as **estimated**, not authoritative
  — the markdown export wording makes that explicit.

## Honest caveats (to surface in output)

- The model assumes the agent would have **succeeded** via `grep`.
  Pre-fix monorepo-4 had an agent that flailed with grep and
  re-tried; counterfactual would *under*-count its real cost.
- `read_focused` with empty focus = `Read` → counterfactual savings
  is 0 (correct, because there's no real saving when focus is empty).
- Graph tools (`get_neighborhood`, `impact_analysis`) have the
  weakest model; their `counterfactual_bytes` carries a flag
  `estimate_quality: low` so aggregators can downweight.

## What this lets the user do

1. **Trend the savings ratio over time.** If A3 lands and over the
   next month the ratio dips below 1.0 (myco costs more than the
   modelled alternative), that's a real signal — possibly a sign of
   the G2 adoption issue (read_focused without focus) inflating
   myco-side bytes.
2. **Validate v3.x adoption work.** When we ship Block B (adoption
   fixed-point), the savings ratio should climb. Without this
   measurement we'd just be guessing.
3. **Frame v4 priorities.** If `get_neighborhood` shows the largest
   per-call savings, federation matters more than graph-DB-rewrite.
   If `read_focused` dominates, the v4 push should be deeper
   focused-read tooling.

## Out of scope

- A live `--bench` mode that actually executes the fallback grep. v4.
- Per-conversation comparison harness (run a whole task twice, with
  and without myco). v4 too — costly to implement, separate workflow
  from per-call telemetry.
- Adjusting the model for non-Go/TS codebases. The 120-B/600-B
  constants are TS+Go-tuned; calibration test will alert when
  someone runs A3 on a Python repo and the gap is large.
